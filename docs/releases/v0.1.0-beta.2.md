# v0.1.0-beta.2 - Packaged Lua Boundary Cleanup

`v0.1.0-beta.2` makes `zlua-portable` behave like a real reusable Zig package
instead of a helper crate that still assumes consumer-specific build hacks.

## Summary

- the package now depends on `ziglua` directly for its Lua C bindings
- package tests now validate against the packaged Lua artifact instead of
  forcing a system `lua5.4` pkg-config dependency
- consumer guidance now reflects the package-managed dependency model

## Why This Release Exists

The previous release still leaked one bad assumption into consumers:

- normal package use was fine
- but package-local testing and some macOS-native bring-up flows still dragged
  a forced `lua5.4` system lookup into the graph

That was acceptable for an early shared-package checkpoint, but not good enough
for native macOS release prep in downstream apps.

## Technical Changes

- `build.zig.zon` now declares the package's `ziglua` dependency explicitly
- `build.zig` wires the generated `ziglua-c` module into `zlua-portable`
- package tests now link the packaged Lua artifact instead of calling
  `linkSystemLibrary("lua5.4", ...)`
- `src/api.zig` now consumes the imported C bindings module rather than
  re-importing Lua headers directly

## Validation

- `zig build test`

## Consumption

Consumers should prefer the pinned package tag path again after updating the
saved hash in their `build.zig.zon`.

Sibling local path overrides remain appropriate only for active co-development.
